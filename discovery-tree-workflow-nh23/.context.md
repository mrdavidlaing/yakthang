## What It Provides

**Just-in-time hierarchical planning:**

- Start simple (don't overplan)
- Break down only what's needed next
- Emergent complexity handling
- Epic → Root Task → Subtasks structure
- Visual status tracking via tree

## Why It's Valuable for Yakthang

- **Already doing this!** yakthang uses yx's hierarchical task structure
- Formalizing the pattern makes it repeatable
- Prevents over-planning (waterfall yaks)
- Natural fit for distributed worker model

## Adaptation Ideas

### For Orchestrator (Yakob)

1. **Epic as top-level yak**
   - Example: `yakthang-v2` is an epic
   - Contains multiple related initiatives

2. **Root tasks as immediate goals**
   - Example: `yakthang-v2/compile-yx`, `yakthang-v2/add-overview-docs`
   - Represent chunks of work that can be assigned to workers

3. **Subtasks for discovered complexity**
   - Example: `yakthang-v2/yak-box/cleanup-zellij-tabs`
   - Created when a root task reveals sub-problems

4. **Just-in-time breakdown**
   - Don't create all subtasks up front
   - Worker discovers them during implementation
   - Worker adds new yaks as blockers are found

### Pattern

```
yx add product-feature                    # Epic
yx add product-feature/backend            # Root task (assign to worker)
# Worker starts work, discovers complexity:
yx add product-feature/backend/auth       # Subtask
yx add product-feature/backend/db-schema  # Subtask
# New root task emerges:
yx add product-feature/frontend           # Root task (different worker)
```

### Integration Points

- **With yak-mapping:** map reveals when breakdown is needed (too vague = needs subtasks)
- **With preparing-a-yak:** root tasks get prepared before spawning; subtasks emerge during work
- **With parallel-yak-implementation:** leaves of the tree can be parallelized
