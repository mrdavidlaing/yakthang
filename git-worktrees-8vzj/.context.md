Integrate git worktrees into worker spawning so each worker can operate in
its own isolated checkout. This avoids file conflicts when multiple workers
edit the same repo concurrently.

## Design

The orchestrator manages the worktree lifecycle:
1. Before spawning a worker, create a git worktree (branch + directory)
2. Pass the worktree path as --cwd to spawn-worker.sh
3. After the worker finishes (task done), the orchestrator cleans up the worktree

From the worker's perspective, nothing changes — they just see a --cwd path
and work in it. They don't know or care that it's a worktree.

## Orchestrator responsibilities
- Create worktree: `git worktree add <path> -b <branch-name>`
- Naming convention for branches: e.g. `worker/<task-name>` or `yak/<task-name>`
- Naming convention for worktree dirs: e.g. `.worktrees/<task-name>/`
- Clean up after worker completes: `git worktree remove <path>`
- Merge or cherry-pick completed work back to main branch
- Handle conflicts if two workers touched overlapping files

## spawn-worker.sh changes
- Optional --worktree flag that triggers worktree creation before spawn
- Or: keep spawn-worker.sh unchanged, have orchestrator create worktree
  first and just pass --cwd <worktree-path> (simpler, preferred)

## Open questions
- Where do worktree directories live? (.worktrees/ at repo root? /tmp?)
- Branch naming convention?
- Merge strategy — auto-merge? PR-style review? Orchestrator decides?
- What about .yaks/ state — it's shared across all worktrees (good, it's
  at the repo root and worktrees share the same .git)
- Does the worktree need the same nix/direnv environment?

## Acceptance criteria
- Orchestrator can create a worktree for a task
- Worker spawns into the worktree and works normally
- Orchestrator can clean up worktrees after completion
- .yaks/ state remains accessible from worktrees
- Document the merge/integration workflow
