EVERGREEN: yak-map (Rust/WASM) code quality improvement

This is a recurring nightshift yak. Each session, Yakob triages potential
improvements, creates sub-yaks, and spawns a shaver to work through them.

## Project details
- CWD: src/yak-map
- Language: Rust, compiled to wasm32-wasip1
- Quality gate: bin/dev check (WASM build + clippy + rustfmt + tests + complexity)
- Complexity tool: bin/dev cx (needs rust-code-analysis-cli via nix)
- Desloppify state: src/yak-map/.desloppify/state-rust.json

## Nightshift workflow

### 1. Discovery (interactive, with operator)

Run quality checks and review results together:

```bash
cd src/yak-map
bin/dev check          # Full quality gate — note any failures
bin/dev cx             # Complexity report — find hotspots
```

Also check desloppify if state file exists:
```bash
cat .desloppify/state-rust.json | jq '.issues | to_entries[] | .value | select(.status == "open") | {id, summary, tier}'
```

Look for:
- Lint warnings or fmt failures (quick wins)
- Functions approaching complexity thresholds (headroom < 3 points)
- Large files (>400 LOC) that could be split
- Desloppify issues by tier (tier 1 = highest priority)

### 2. Curate (interactive, with operator)

Present findings as a menu. For each candidate improvement:
- Estimate risk (low/medium/high)
- Note if it's safe for unattended nightshift

Operator picks which ones go into tonight's batch.

### 3. Create sub-yaks

For each selected improvement, create a child yak with specific context:

```bash
yx add <improvement-name> --under improve yak-map
echo '<specific context with files, approach, definition of done>' | yx context <improvement-name>
```

Context must include:
- What to change and where (file paths)
- How to verify (bin/dev check must pass)
- Risk level and any constraints

### 4. Spawn nightshift shaver

One shaver, all sub-yaks, in a worktree:

```bash
yak-box spawn \
  --cwd ./src/yak-map \
  --yak-name "yak-map-quality" \
  --shaver-name "<name>" \
  --tool claude --runtime native \
  --auto-worktree \
  --yaks <yak-paths...> \
  "<prompt with ordered yak list, verification steps, PR creation>"
```

Key shaver instructions:
- Work yaks sequentially in the specified order
- Run bin/dev check after EVERY change
- One commit per yak (use /yak-brand)
- Create PR when all yaks done
- Do not proceed if checks fail

### 5. Morning review

- Review PR with difit: `difit --pr <url> --no-open`
- Merge or request changes
- Mark done yaks, prune, wrap
