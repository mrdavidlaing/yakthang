Include a 'validation command' field in task context written by Yakob.

## Problem
The Claude Code Insights report shows config fix cycles taking 3+ rounds because
shavers fix one error, declare done, then discover more errors on the next run.
This appeared across Hyprland, Nix, YAML, and docker-compose sessions.

## Desired Changes
When Yakob writes task context (via 'yx context'), include a validation command
that the shaver must run before marking the task done. Examples:
- Nix tasks: 'nix flake check'
- Rust tasks: 'cargo build && cargo test && cargo clippy'
- YAML/config tasks: 'yamllint <file>' or the relevant reload command
- Docker tasks: 'docker-compose config --quiet'

## Implementation Options
1. Add a 'validation-command' field to yaks (yx field <id> validation-command)
   that the shaver reads and runs before marking done
2. Update the yak-shaving-handbook skill: 'Before running yx done, check if
   a validation-command field exists and run it. Fix ALL errors, not just the first.'
3. Update Yakob's task context template to always include this field

## Acceptance Criteria
- Every task with a code/config change has a validation command
- Shavers run validation before marking done
- The 'fix one error per round' anti-pattern is eliminated
