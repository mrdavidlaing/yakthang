## Background

Native spawn currently sets CLAUDE_CONFIG_DIR=<workerDir>/.claude and calls
setupClaudeSettings() to pre-seed skills, settings.json (apiKeyHelper), and
.claude.json. This was introduced to replace the original HOME=<workerDir>
override (which caused macOS keychain dialogs).

The question is whether even this CLAUDE_CONFIG_DIR isolation is worth it for
native mode, given that native is explicitly the "maximum flexibility /
trust the shaver" runtime.

## What CLAUDE_CONFIG_DIR currently provides

1. Per-worker skills dir — workers get yak-box skills in ~/.claude/skills/.
   Counter: skills are already injected inline into the prompt. If the user has
   yak-box skills in their own ~/.claude/skills/ too, that's fine.

2. apiKeyHelper in settings.json — api-key-helper.sh echoes _ANTHROPIC_API_KEY.
   Counter: Claude Code reads ANTHROPIC_API_KEY natively. Could set it directly
   and skip the helper+settings.json write entirely.

3. Pre-seeded .claude.json — skips onboarding, bypasses permissions prompt.
   Counter: any real operator already has hasCompletedOnboarding:true in their
   own ~/.claude.json. The pre-seed only matters on a fresh machine.

4. Session separation — worker sessions don't appear in the operator's /resume.
   Counter: mild UX nuisance at worst. Could be a feature (operator can see
   what workers did).

## Pros of stripping

- No setup machinery in native spawn — wrapper script becomes trivially simple
- Workers use the operator's real ~/.claude/ — auth, settings, skills just work
- No clobbering of the operator's settings.json
- No per-worker .claude/ dir to manage / clean up
- Fewer moving parts = fewer failure modes

## Cons of stripping

- Worker sessions show up in the operator's /resume list (noise)
- Workers share operator's Claude session history
- No per-worker skill isolation (workers see operator's skills, not just yak-box
  skills) — probably fine but untested
- If two workers run concurrently and both write to ~/.claude/ simultaneously
  there could be write conflicts on settings.json or .claude.json

## Proposed implementation

In SpawnNativeWorker:
- Remove call to setupClaudeSettings()
- Remove export CLAUDE_CONFIG_DIR from wrapper script
- Replace _ANTHROPIC_API_KEY + apiKeyHelper pattern with ANTHROPIC_API_KEY
  set directly in the wrapper env

Keep:
- YAK_PATH, YAK_SHAVER_NAME, IS_DEMO (still needed)
- MODEL passthrough
- unset CLAUDECODE (still needed to avoid nested session conflicts)
- PID file write

## Files to change

- src/yak-box/internal/runtime/native.go — generateNativeWrapperScript (claude case)
  and SpawnNativeWorker (remove setupClaudeSettings call)
- src/yak-box/internal/runtime/native_test.go — update wrapper content assertions
- src/yak-box/internal/runtime/helpers_test.go — update wrapper content assertions

## Definition of Done

- Native claude wrapper no longer sets CLAUDE_CONFIG_DIR
- Native claude wrapper no longer calls setupClaudeSettings
- API key passed as ANTHROPIC_API_KEY (not via helper script)
- All tests pass
- Manual test: spawn a native worker, verify it authenticates and runs correctly
