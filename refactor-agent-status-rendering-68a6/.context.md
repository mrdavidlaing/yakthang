Rethink how agent-status drives yak-map rendering (color + symbol).

## Current State
AgentStatusKind enum (Blocked/Done/Wip/Unknown) parses the prefix from agent-status strings. task_color() and status_symbol() each check agent-status first, then fall back to TaskState. This creates a two-layer priority system where agent-status overrides TaskState for visual rendering.

## Questions to Resolve
- Should agent-status and TaskState be unified into a single resolved display state, rather than two separate priority checks in each rendering function?
- The Done-via-agent-status (green) vs Done-via-TaskState (dim) distinction is implicit — should this be made explicit in a type?
- Would a single "resolve visual state" function be cleaner than sprinkling the priority logic across task_color() and status_symbol()?
- Consider whether review_status (pass/fail/in-progress) should also participate in a unified rendering model.

## Definition of Done
- Document the rendering priority logic clearly (which status source wins, and why)
- Refactor so the priority resolution happens in one place, not duplicated across rendering functions
- All existing tests pass
- No visual behavior changes (this is a refactor, not a feature change)
