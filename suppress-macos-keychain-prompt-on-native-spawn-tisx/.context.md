Rethink how yak-box native runtime uses the HOME folder.

## Root Problem
The native runtime resets HOME to the worker directory (.yak-boxes/<worker>/@home). This breaks many macOS assumptions:
- macOS keychain expects HOME to contain ~/Library/Keychains — hence the dialog
- Claude Code cannot find itself in PATH after HOME is changed (even though ~/.local/bin is prepended, there may be other Claude Code bootstrap paths that rely on the original HOME)
- Potentially other tools (git credential helpers, npm, etc.) may also break

## Current Workarounds (That Aren't Working Well)
- PATH prepend: `export PATH="$hostHome/.local/bin:$PATH"` before overriding HOME
- GIT_CONFIG_GLOBAL and GH_CONFIG_DIR explicitly pointed back at host home
- macOS keychain: create a worker-local keychain, set as default, restore on exit
These are whack-a-mole patches. Each new tool that relies on HOME requires another workaround.

## Key Question
**Should we be changing HOME at all for native spawns?**

The purpose of the worker @home dir is to give each worker an isolated Claude Code config (~/.claude/) so workers don't conflict with each other or with the user's interactive session. But we don't need to redirect *all* of HOME to achieve that — we only need Claude Code's config directory to be isolated.

## Approach to Explore
Instead of `export HOME=<worker-dir>`, consider:
- Keep HOME as the real user home
- Use `export CLAUDE_CONFIG_DIR=<worker-dir>/.claude` (or equivalent Claude Code env var) to redirect only Claude's config
- Keep all macOS/system tooling pointing at real HOME — no breakage

## What to Investigate
1. Does Claude Code support a config dir env var? Check claude --help, Claude Code docs, or source
   - Look for: CLAUDE_CONFIG_DIR, CLAUDE_HOME, XDG_CONFIG_HOME, or similar
2. What does the worker @home dir actually contain today? (List .yak-boxes/<any-worker>/@home)
3. Which parts of the worker @home are Claude-specific vs generic?
4. If CLAUDE_CONFIG_DIR exists: prototype the change in native.go — remove HOME override, use config dir var instead, drop the keychain workaround
5. If no such env var exists: document what options we have (e.g. symlink, XDG, patching)

## Related Symptom
Claude Code reports it cannot find itself in PATH after spawn. This is likely a symptom of HOME being changed (node_modules/.bin or similar resolving relative to HOME). Document whether the proposed fix also resolves this.
