## What It Provides

The **Approach Pattern** for emergent planning:
1. Add one yak toward your goal
2. **Show map** (critical step - always do this after adding)
3. Add context explaining the yak
4. Explore dependencies and blockers

**Context structure:**
- Goal: what you're trying to achieve
- Definition of Done: how you'll know it's complete
- Known Knowns: facts you're confident about
- Known Unknowns: questions that need answers

## Why It's Valuable for Yakthang

- **Core orchestrator pattern** — how Yakob plans work
- Emergent discovery vs. big up-front design
- Forces visibility into dependencies
- Already using `yx ls` for mapping

## Adaptation Ideas

### For Orchestrator (Yakob)

1. **Formalize the mapping cadence**
   - `yx add <yak>` → `yx ls` → `yx context <yak>`
   - Make this a reflex, not an afterthought

2. **Enhance visualization**
   - Could integrate with yak-map Zellij plugin
   - Show worker assignments in the map view

3. **Context template**
   - Create a standard context.md template with Goal/DoD/Known Knowns/Known Unknowns sections
   - Script to scaffold this when adding yaks

### For Workers

1. **Pre-work check**
   - Before starting implementation, show the map
   - Verify dependencies are clear

2. **Discovery notes**
   - Workers add findings back to context during work
   - Update Known Unknowns → Known Knowns as discoveries happen

### Integration Points

- **With preparing-a-yak:** yak-mapping discovers what needs preparation
- **With discovery-tree-workflow:** map shows when to break down further
- **With parallel-yak-implementation:** map reveals independent leaves ready for parallel work
