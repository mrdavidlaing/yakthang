# Rust Code Quality: yak-map Improvements Inspired by yx

## Current State

**yak-map**: 1295-line monolith in `src/yak-map/src/main.rs`. Zellij plugin that renders the yak DAG as a tree. ~52% tests (675 lines). No panics, safe for WASM. But all concerns live in one file with no module boundaries.

**yx**: Clean hexagonal architecture across domain/, application/, adapters/, infrastructure/. Event-sourced with CRDT merge. Compiler-enforced architectural boundaries via traits. Strong type system usage.

---

## QUICK WINS (low effort, high readability impact)

### QW-1: Extract color/rendering constants
**Effort**: ~30 min | **Impact**: Readability
**Problem**: ANSI codes like `"\x1b[31m"` scattered across `task_color()`, `render_task()`, `tree_prefix()`, `highlight_line()` (lines 343-453).
**yx pattern**: Not directly — but yx uses structured display via `DisplayPort` trait (`domain/ports/user_display.rs`).
**Fix**: Create a `colors` module or const block. Replace magic numbers with named constants:
```rust
const RED: &str = "\x1b[31m";
const RESET: &str = "\x1b[0m";
const BG_SELECTED: u8 = 237;
```
**Files**: `src/yak-map/src/main.rs` lines 340-411

### QW-2: Extract utility functions to a module
**Effort**: ~30 min | **Impact**: File size, separation of concerns
**Problem**: `escape_single_quoted()` (169-182), `copy_via_zellij_tty()` (187-204), `base64_encode()` (206-220), `strip_ansi()` (229-246), `line_display_width()` (224-226) are standalone utilities mixed into the main file.
**yx pattern**: yx separates infrastructure concerns (git_discovery.rs) from domain logic.
**Fix**: Move to `src/yak-map/src/util.rs`. These have zero dependencies on State or TaskRepository.
**Files**: `src/yak-map/src/main.rs` lines 154-246

### QW-3: Replace custom base64 with crate
**Effort**: ~15 min | **Impact**: Code debt reduction
**Problem**: 15-line custom base64 encoder (lines 206-220) reimplements what `base64` crate provides.
**Fix**: Add `base64` to Cargo.toml, replace `base64_encode()` with `base64::engine::general_purpose::STANDARD.encode()`. Or keep it if WASM compat is a concern — but document why.
**Files**: `src/yak-map/src/main.rs` lines 206-220, `src/yak-map/Cargo.toml`

### QW-4: Remove `#![allow(unused)]`
**Effort**: ~15 min | **Impact**: Code hygiene
**Problem**: Line 1 suppresses all unused warnings globally. May be hiding dead code.
**Fix**: Remove the directive, fix any actual unused warnings, understand what was being suppressed.
**Files**: `src/yak-map/src/main.rs` line 1

### QW-5: Extract agent-status prefix matching
**Effort**: ~20 min | **Impact**: DRY, consistency
**Problem**: Agent status prefix matching (`blocked:`, `done:`, `wip:`) is repeated across `task_color()` (lines 341-350) and `status_symbol()` (lines 359-366).
**yx pattern**: yx uses `YakState` enum with `FromStr` impl (`domain/yak_state.rs` lines 25-39) — parsing happens once, type carries meaning.
**Fix**: Create an `AgentStatusKind` enum parsed from the status string. Both functions consume the enum instead of re-parsing.
**Files**: `src/yak-map/src/main.rs` lines 340-373

## MEDIUM REFACTORS (1-2 hours each, significant quality improvement)

### MR-1: Extract rendering module
**Effort**: ~1-2 hours | **Impact**: Separation of concerns, testability
**Problem**: `task_color()`, `status_symbol()`, `tree_prefix()`, `highlight_line()`, `render_task()`, `review_status_emoji()` are all rendering functions (~150 lines) mixed with business logic.
**yx pattern**: yx has dedicated `DisplayPort` trait (`domain/ports/user_display.rs`) with multiple implementations (TuiDisplay, ConsoleDisplay, JsonDisplay). Rendering is a separate adapter.
**Fix**: Create `src/yak-map/src/render.rs`. Move all rendering functions there. State calls `render::render_task(&task_line, is_selected)` etc. Tests for rendering move too.
**Files**: `src/yak-map/src/main.rs` lines 156-167, 340-453

### MR-2: Extract tree builder from refresh_tasks()
**Effort**: ~1-2 hours | **Impact**: Readability (biggest single win)
**Problem**: `refresh_tasks()` is 90 lines (249-338) doing 4 things: collecting tasks, computing children maps, computing sibling order, computing ancestor continuation tracking. The nested loop at lines 290-331 is the hardest code to follow in the file.
**yx pattern**: yx's `YakMap` aggregate (`domain/yak_map.rs`) encapsulates tree operations. `find_children_of()`, `get_ancestor_ids()` are separate methods on the domain model.
**Fix**: Create a `TreeBuilder` struct or module that takes raw tasks and produces a `Vec<TaskLine>` with all tree metadata computed. `refresh_tasks()` becomes: `self.tasks = TreeBuilder::build(&repo)`.
**Files**: `src/yak-map/src/main.rs` lines 249-338

### MR-3: Break up State.update() event handler
**Effort**: ~1 hour | **Impact**: Readability, testability
**Problem**: `update()` (lines 500-572) is a 73-line triple-nested match handling 6+ key events, file creation, clipboard, and navigation — all in one method.
**yx pattern**: yx uses the UseCase pattern (`application/use_case.rs`). Each command is a separate struct with `execute()`. CLI routing in main.rs just dispatches to the right use case.
**Fix**: Extract handlers: `handle_navigation(&mut self, key)`, `handle_edit(&mut self)`, `handle_copy(&mut self)`, `handle_show(&mut self)`. The `update()` match becomes a dispatcher.
**Files**: `src/yak-map/src/main.rs` lines 500-572

### MR-4: Introduce TaskRepository trait (port)
**Effort**: ~1-2 hours | **Impact**: Testability
**Problem**: `TaskRepository` is a concrete struct that reads from the filesystem. Tests use `tempfile::TempDir` to create real directories. This works but is slower and more fragile than in-memory testing.
**yx pattern**: yx defines `ReadYakStore` and `WriteYakStore` traits (`domain/ports/yak_store.rs`), with `InMemoryStorage` for testing and `DirectoryStorage` for production. Contract tests (`yak_store/contract_tests.rs`) verify both implementations behave identically.
**Fix**: Extract a `TaskSource` trait with `list_tasks()`, `get_field()`, `get_task()`. Create `InMemoryTaskSource` for tests. State takes `Box<dyn TaskSource>`.
**Files**: `src/yak-map/src/main.rs` lines 15-107, test helpers at 620-640

## LARGER REFACTORS (half-day+, architectural improvement)

### LR-1: Multi-file module structure
**Effort**: ~half day | **Impact**: Maintainability, navigation
**Problem**: 1295 lines in one file. Every change requires navigating a wall of code.
**yx pattern**: yx has clear module hierarchy: `domain/` (models, ports, events), `application/` (use cases), `adapters/` (implementations), `infrastructure/` (cross-cutting).
**Fix**: Split into:
```
src/yak-map/src/
├── main.rs        (~100 lines: ZellijPlugin impl, load/update/render dispatch)
├── lib.rs         (module declarations)
├── model.rs       (TaskState, TaskLine — domain types)
├── repository.rs  (TaskRepository — filesystem adapter)
├── tree.rs        (TreeBuilder — hierarchy computation)
├── render.rs      (Colors, symbols, tree prefixes, task rendering)
├── clipboard.rs   (OSC 52, shell escaping)
└── util.rs        (ANSI stripping, base64, display width)
```
**Prerequisite**: QW-1, QW-2, MR-1, MR-2 — do these first and LR-1 becomes a simple file split.
**Files**: `src/yak-map/src/main.rs` (entire file)

### LR-2: Type-safe domain modeling
**Effort**: ~half day | **Impact**: Correctness, expressiveness
**Problem**: Tasks are represented as `TaskLine` with `String` fields: name, id, path are all plain strings. State is a string parsed repeatedly.
**yx pattern**: yx uses newtype wrappers: `YakId` (deterministic hash-based ID), `Slug` (filesystem-safe name), `Name` (display name), `YakState` (enum). Each has its own validation and `FromStr`/`Display` impls (`domain/slug.rs` lines 3-124).
**Fix**: Introduce `TaskId`, `TaskName`, `TaskPath` newtypes. Parse `TaskState` once at load time (already an enum — good). Add validation at construction boundaries.
**Note**: Lower priority than structural refactors — yak-map is a read-only view, so type confusion has less blast radius than in yx.

### LR-3: Structured error handling
**Effort**: ~2-3 hours | **Impact**: Debuggability
**Problem**: Errors are silently swallowed with `.ok()` (~10 instances) and `let _ = fs::write()`. No logging, no user feedback on failures.
**yx pattern**: yx uses `anyhow::Result<T>` throughout with `?` propagation. Validation errors carry context. Domain errors are typed.
**Fix**: Define a `YakMapError` enum or use `anyhow`. Propagate errors to `render()` for display in the error toast (line 581 — infrastructure already exists: `self.error`). At minimum, log failures to the existing `self.error` field so they're visible.
**Files**: `src/yak-map/src/main.rs` lines 38, 45, 66, 535, 538

## ADDITIONAL PATTERNS FROM YX WORTH ADOPTING

### AP-1: Inner + Outer Loop Testing
**yx pattern**: Inner loop = `#[test]` unit tests + in-process cucumber (~1s). Outer loop = full-stack cucumber spawning the binary (~39s) + shellspec. Both cucumber worlds implement a shared `TestWorld` trait so step definitions are written once.
**yak-map opportunity**: Co-locate tests with extracted modules (after MR-1, MR-2). After MR-4 (TaskRepository trait), add contract tests (yx's `yak_store_tests!` macro pattern) to verify InMemoryTaskSource and real TaskRepository behave identically.

### AP-2: bin/dev Quality Gate
**yx pattern**: `src/yaks/bin/dev` (978 lines) runs: clippy + rustfmt + shellcheck → complexity analysis (cognitive ≤ 42, cyclomatic ≤ 34) → cargo audit → mutation testing on changed files. `dev check` runs everything in sequence.
**yak-map opportunity**: Create `src/yak-map/bin/dev` or add yak-map targets to the existing one. The complexity thresholds alone would flag `refresh_tasks()` (90 lines) and `update()` (73 lines). Mutation testing (`cargo mutants`) would reveal untested paths in the event handler.

### AP-3: Preparing-a-Yak Workflow
**yx pattern**: `src/yaks/skills/preparing-a-yak/SKILL.md` — structured interview that takes a vague idea through brainstorming → example mapping (for features) or ADR (for refactorings) → sub-yak planning. Outputs stored via `yx context` and `yx field`.
**yak-map opportunity**: Each MR/LR item in this plan could go through this preparation flow before implementation. Ensures definition of done is concrete and testable.

### AP-4: Mutation Test → Yak Pipeline
**yx pattern**: `dev mutate-sync` reads `mutants.out/missed.txt` and auto-creates yaks for missed mutants under a parent yak. Reopens done yaks if new misses appear. A self-improving quality loop.
**yak-map opportunity**: Once yak-map has a `bin/dev` with mutation testing, this pipeline would auto-generate improvement tasks for untested code paths.

## CAPTURING ARCHITECTURAL INTENT: ADRs vs Project Skills

Two complementary mechanisms for ensuring code quality persists:

### ADRs (Architecture Decision Records)
**What they capture**: The *decision and its rationale* — "we chose to extract a TreeBuilder because X; we considered inlining the logic but rejected it because Y."
**yx pattern**: Stored as fields on yaks (`yx field <name> adr`). Used by the preparing-a-yak skill for refactoring-type yaks.
**Nature**: Historical records. They explain *why* the codebase looks the way it does. Useful for onboarding and for revisiting decisions when constraints change.

### Project Skills
**What they capture**: The *living convention* — "when writing code in yak-map, follow these patterns."
**Nature**: Active instructions that shape future work. Every agent spawned against the codebase absorbs these conventions automatically.

### Recommendation: Use Both
- **Write an ADR** when making an architectural decision (e.g., "why we extracted TreeBuilder", "why we chose trait-based ports over concrete types for testing").
- **Encode the resulting convention as a project skill** (e.g., a `yak-map-architecture` skill that says: "rendering functions go in render.rs", "use TreeBuilder for hierarchy computation", "all color constants live in the colors module", "prefer the TaskSource trait over direct filesystem access in tests").

The ADR answers "why did we decide this?" — useful when revisiting or evolving the architecture. The skill answers "what do I do now?" — useful for every agent touching the code going forward.

A `yak-map-architecture` skill could encode:
- Module responsibilities and file boundaries
- Naming conventions for functions and types
- Testing patterns (which adapter to use, where tests live)
- Error handling conventions
- Rendering patterns (how colors and formatting work)

This skill would evolve alongside the codebase as refactors land, serving as the living architectural guide that keeps agents aligned with intent.

## APPROACH RECOMMENDATION

Do items 1-4 (quick wins) as a single commit. They're independent, low-risk, and immediately improve the file.

Then tackle MR-2 (tree builder) — it's the single biggest readability improvement and the code that's hardest to maintain today.

MR-1 and MR-3 follow naturally. After those three medium refactors, LR-1 (multi-file split) becomes trivial — just moving already-separated code into files.

Skip LR-2 (type-safe modeling) unless yak-map starts doing writes. It's a read-only view; string types are adequate.

As each refactor lands, write an ADR capturing the decision, then update a `yak-map-architecture` project skill with the resulting convention. This ensures both the rationale and the practice are preserved for future agents.
