Study src/yaks (yx) Rust codebase for code quality patterns to apply to yak-map.

## The Opportunity

yak-map (src/yak-map/) is a 1295-line monolith in a single main.rs.
src/yaks has a clean hexagonal architecture:
  - domain/ (ports, yak_map model)
  - application/ (use cases: add_yak, prune_yaks, etc.)
  - adapters/ (directory storage, user input, local workspace)
  - infrastructure/ (git discovery)

## What to Look For

- How yx structures ports and adapters (trait definitions, impl separation)
- Testing patterns (in-process vs full-stack test worlds)
- Error handling idioms
- Module organization and public API boundaries
- Use of Rust type system for domain modeling
- How context/state flows through the application layer

## Goal

Produce a concrete list of improvements for yak-map, prioritized by
impact vs effort. Not a full rewrite — incremental improvements that
bring yak-map closer to yx quality standards.

## Key Files

- yak-map source: src/yak-map/src/main.rs (1295 lines, single file)
- yak-map build: src/yak-map/build.rs
- yx architecture: src/yaks/src/ (domain/, application/, adapters/, infrastructure/)
- yx main: src/yaks/src/main.rs
- yx lib: src/yaks/src/lib.rs

## Definition of Done

- Documented list of specific, actionable improvements for yak-map
- Each improvement references a concrete pattern from src/yaks
- Improvements ranked by impact (readability, testability, maintainability)
- Quick wins separated from larger refactors
