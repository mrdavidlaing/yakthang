Prevent heartbeat loops from running unbounded and wasting resources.

## Problem
The Claude Code Insights report documents an 18-hour heartbeat loop running 150+
no-op 'yx ls' calls. The /loop 5m yx ls pattern Yakob uses has no termination
condition — it runs until the session dies or the human kills it.

The yak-nightshift skill already has a fix for cancelling when the queue is
exhausted (commit e2fcff9), but the general /loop mechanism needs hardening.

## Desired Behavior
1. Heartbeat loops respect the session hard-stop time and self-terminate
2. Diff-based: only report when yx ls output actually changes (hash comparison)
3. Max-duration cap: loops should not exceed the session time window
4. Max-iterations cap: safety valve (e.g., 100 iterations max)

## Possible Approaches
- Enhance the /loop skill to accept --until and --max-iterations flags
- Build diff-detection into yx itself (yx watch --on-change <cmd>)
- Have Yakob's heartbeat use a wrapper script that checks session hard-stop

## Acceptance Criteria
- No heartbeat loop runs longer than the session hard-stop
- Idle loops (no changes detected) self-terminate after N consecutive no-ops
- The 18-hour runaway scenario is impossible
