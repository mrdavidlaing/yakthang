## What It Provides

**Three-phase requirements process:**

1. **Brainstorm** — diverge on possibilities
2. **Example Map** — concrete scenarios with inputs/outputs  
3. **Implementation Plan** — steps and approach

Stores outputs using `yx field` commands on the yak itself.

## Why It's Valuable for Yakthang

- **Critical orchestrator responsibility** — prepare clear work before spawning workers
- Reduces worker blocking due to unclear requirements
- Creates shared understanding before delegation
- Already using `yx context` for text; can extend with fields

## Adaptation Ideas

### For Orchestrator (Yakob)

1. **Pre-spawn checklist**
   - Before `yak-box spawn`, verify yak has:
     - Clear goal statement
     - Definition of Done
     - Context with examples (if applicable)
     - Known constraints/dependencies

2. **Use yx fields**
   - `yx field <yak> goal` — one-line statement
   - `yx field <yak> dod` — acceptance criteria
   - `yx field <yak> examples` — concrete scenarios
   - `yx field <yak> plan` — implementation approach
   - Keep `context.md` for free-form notes

3. **Brainstorming mode**
   - When adding a complex yak, spend time exploring approaches
   - Document alternatives considered (and why chosen path was selected)

### For Workers

1. **Read before starting**
   - Workers check `yx field --show <yak> goal dod examples plan`
   - If unclear, set status to `blocked: needs clarification`

2. **Update during work**
   - Workers add discoveries to plan field
   - If assumptions were wrong, document what changed

### Integration Points

- **With yak-mapping:** mapping reveals what needs preparation
- **With discovery-tree-workflow:** preparation happens before breaking down
- **With parallel-yak-implementation:** well-prepared yaks are easier to parallelize
