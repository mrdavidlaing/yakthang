## Problem
When `yak-box stop --yak-name <worker>` runs, it tries to clear the `assigned-to` field from the yak's task directory. But it consistently fails with:

    Warning: Failed to find task directory for <yak-name>: no directory matching "<slug>" found under <path>/.yaks

This means `assigned-to` is never cleared on stop, requiring manual cleanup every time.

## Evidence from today's session (2026-03-14)
Every single `yak-box stop` failed to clear assignments:
- `yak-box stop --yak-name "agent-status-enum"` → "no directory matching agent-status-kind-enum"
- `yak-box stop --yak-name "extract-utils"` → "no directory matching extract-utilities-module"  
- `yak-box stop --yak-name "review-status-enum"` → "no directory matching review-status-kind-enum"
- `yak-box stop --yak-name "task-state-fromstr"` → "no directory matching task-state-from-str"
- `yak-box stop --yak-name "extract-render"` → "no directory matching extract-rendering-module"

The pattern: the yak-name (tab name) doesn't match the yak's directory slug under .yaks/. The stop command seems to be searching by the worker name or a derivative, not the actual yak ID or task name.

## Root Cause Investigation
Look at how `yak-box stop` resolves task directories. It needs to either:
1. Use the yak IDs stored in the session (from --yaks flags at spawn time)
2. Read the session metadata to find which yaks were assigned
3. Search .yaks/ by assigned-to field value instead of name matching

## Where to Look
- yak-box stop command implementation (likely src/yak-box/cmd/stop.go or similar)
- Session storage: .yak-boxes/sessions.json
- How --yaks flags are stored at spawn time

## Definition of Done
- `yak-box stop` successfully clears assigned-to on the yak task directories
- Works even when yak-name differs from the yak's directory slug
- No more "Failed to find task directory" warnings during normal operation
- Test with a spawn/stop cycle to verify
