## Executive Summary

This document provides a detailed implementation plan for restricting network access of Docker workers to only essential LLM API endpoints. The current setup uses `--network bridge`, which grants full internet access—allowing workers to potentially exfiltrate data, clone arbitrary repositories, or reach unintended services.

**Goal:** Implement network filtering that:
- Allows HTTPS access to specified LLM API endpoints only
- Blocks all other outbound connections
- Maintains compatibility with spawn-worker.sh
- Is maintainable and auditable
- Works with the existing container hardening (read-only, non-root, tmpfs mounts)

**Recommended Solution:** **Custom Docker network with iptables filtering** (Option A)

### Security Context

The worker containers already implement strong isolation:
- Read-only root filesystem (`--read-only`)
- All capabilities dropped (`--cap-drop ALL`)
- No privilege escalation (`--security-opt no-new-privileges`)
- Non-root user (`--user $(id -u):$(id -g)`)
- Resource limits (CPU, memory, PIDs)
- Tmpfs-only writable storage

**Network is the remaining attack surface.**

## Solution Options Evaluation

### Option A: Custom Docker Network + iptables (RECOMMENDED)

**Architecture:**
1. Create a custom Docker bridge network: `yak-workers`
2. Configure iptables rules on the host to filter traffic from that network
3. Use DNS resolution to handle dynamic IPs (API endpoints may use CDNs)
4. Apply rules via systemd service or setup script

**Advantages:**
- ✅ **Most flexible** - Can update rules without recreating containers
- ✅ **DNS-aware** - Can filter by domain name using iptables with ipset
- ✅ **Standard approach** - Well-documented Docker networking pattern
- ✅ **Auditable** - Rules visible via `iptables -L` and `docker network inspect`
- ✅ **No proxy overhead** - Direct connections to APIs
- ✅ **Compatible** - Works with existing spawn-worker.sh with minimal changes

**Disadvantages:**
- ⚠️ Requires host-level iptables configuration
- ⚠️ Need to handle DNS resolution and IP updates for CDNs
- ⚠️ Slightly complex setup (but one-time)

**Implementation Complexity:** Medium

---

### Option B: HTTP Proxy (Squid or Tinyproxy)

**Architecture:**
1. Run a lightweight HTTP proxy container (Squid or Tinyproxy)
2. Configure whitelist of allowed domains in proxy config
3. Set `HTTP_PROXY`/`HTTPS_PROXY` environment variables in worker containers
4. Block direct internet access, force all traffic through proxy

**Advantages:**
- ✅ **Simple configuration** - Whitelist is just a text file
- ✅ **Logging** - Can log all connection attempts
- ✅ **DNS-native** - Proxy resolves domains, handles CDN IPs automatically
- ✅ **No host changes** - Runs entirely in Docker

**Disadvantages:**
- ❌ **Additional service** - Proxy container must be maintained
- ❌ **Proxy awareness** - Applications must honor HTTP_PROXY (most do, but not guaranteed)
- ❌ **Performance overhead** - All traffic goes through proxy
- ❌ **Certificate handling** - HTTPS interception is complex (MITM concerns)
- ⚠️ Tinyproxy/Squid don't support HTTPS CONNECT filtering by default without complex configs

**Implementation Complexity:** Medium-High

---

### Option C: nftables Rules (Modern Alternative to iptables)

**Architecture:**
- Same as Option A, but using nftables instead of iptables
- nftables is the modern replacement for iptables in newer Linux kernels

**Advantages:**
- ✅ **Modern** - Better performance than iptables
- ✅ **Simpler syntax** - More intuitive rule syntax
- ✅ Same flexibility as Option A

**Disadvantages:**
- ❌ **Compatibility** - Ubuntu 24.04 supports both, but iptables is more standard
- ❌ **Documentation** - Less Docker-specific documentation for nftables
- ⚠️ Team may be more familiar with iptables

**Implementation Complexity:** Medium

**Note:** This is a valid alternative to Option A. The choice between iptables and nftables is mostly a matter of preference. For this project, **iptables is recommended** due to better documentation and team familiarity.

---

### Option D: Docker Network Plugins (Calico, Weave)

**Architecture:**
- Install a Docker network plugin that provides network policy support
- Define network policies for worker containers

**Advantages:**
- ✅ **Kubernetes-style policies** - Familiar to teams using k8s

**Disadvantages:**
- ❌ **Overkill** - These plugins are designed for clusters, not single-host setups
- ❌ **Complexity** - Significant operational overhead
- ❌ **Dependencies** - Additional services and agents
- ❌ **Not standard** - Would require major changes to deployment

**Implementation Complexity:** Very High

**Verdict:** Not suitable for this use case.

### Implementation Steps

#### Step 1: Create Custom Docker Network

**Script: `/home/yakob/yakthang/scripts/setup-worker-network.sh`**

```bash
#!/usr/bin/env bash
set -euo pipefail

# Create yak-workers network if it doesn't exist
if ! docker network inspect yak-workers >/dev/null 2>&1; then
    docker network create \
        --driver bridge \
        --subnet 172.20.0.0/16 \
        --gateway 172.20.0.1 \
        --opt com.docker.network.bridge.name=br-yak-workers \
        yak-workers
    echo "Created yak-workers Docker network"
else
    echo "yak-workers network already exists"
fi
```

**Why this subnet?**
- `172.20.0.0/16` is in the Docker private IP range
- Avoids conflicts with default Docker bridge (172.17.0.0/16)
- Large enough for many workers (65,534 IPs)

#### Step 2: Configure iptables Rules

Two approaches for handling dynamic IPs:

##### Approach 2A: ipset with DNS updates (RECOMMENDED)

Use `ipset` to create a dynamic set of allowed IPs that can be updated as DNS changes.

**Script: `/home/yakob/yakthang/scripts/update-api-ipset.sh`**

```bash
#!/usr/bin/env bash
set -euo pipefail

# Create ipset if it doesn't exist
if ! ipset list yak-allowed-apis >/dev/null 2>&1; then
    ipset create yak-allowed-apis hash:ip timeout 3600
    echo "Created ipset: yak-allowed-apis"
fi

# Function to add IPs from domain to ipset
add_domain_ips() {
    local domain="$1"
    echo "Resolving $domain..."
    
    # Get IPv4 addresses
    dig +short "$domain" A | grep -E '^[0-9]+\.' | while read -r ip; do
        ipset add yak-allowed-apis "$ip" -exist
        echo "  Added $ip ($domain)"
    done
    
    # Get IPv6 addresses (if needed)
    # dig +short "$domain" AAAA | grep -E '^[0-9a-f:]+' | while read -r ip; do
    #     ipset add yak-allowed-apis "$ip" -exist
    # done
}

# Add all allowed API domains
add_domain_ips "api.anthropic.com"
add_domain_ips "api.openai.com"
add_domain_ips "models.dev"
add_domain_ips "generativelanguage.googleapis.com"

echo "ipset updated successfully"
ipset list yak-allowed-apis | head -20
```

**iptables rules script: `/home/yakob/yakthang/scripts/setup-worker-firewall.sh`**

```bash
#!/usr/bin/env bash
set -euo pipefail

# Ensure ipset is installed
if ! command -v ipset >/dev/null 2>&1; then
    echo "Installing ipset..."
    apt-get update && apt-get install -y ipset
fi

# Source network CIDR
WORKER_NETWORK="172.20.0.0/16"

# Create ipset if needed
if ! ipset list yak-allowed-apis >/dev/null 2>&1; then
    ipset create yak-allowed-apis hash:ip timeout 3600
fi

# Check if rules already exist (idempotent)
if iptables -C FORWARD -s "$WORKER_NETWORK" -j YAK_WORKER_FILTER 2>/dev/null; then
    echo "Firewall rules already configured"
    exit 0
fi

# Create custom chain for worker filtering
iptables -N YAK_WORKER_FILTER 2>/dev/null || true
iptables -F YAK_WORKER_FILTER

# Allow established/related connections (return traffic)
iptables -A YAK_WORKER_FILTER -m conntrack --ctstate ESTABLISHED,RELATED -j ACCEPT

# Allow DNS (required for API hostname resolution)
iptables -A YAK_WORKER_FILTER -p udp --dport 53 -j ACCEPT
iptables -A YAK_WORKER_FILTER -p tcp --dport 53 -j ACCEPT

# Allow HTTPS to whitelisted API IPs
iptables -A YAK_WORKER_FILTER -p tcp --dport 443 -m set --match-set yak-allowed-apis dst -j ACCEPT

# Log denied connections (for monitoring and debugging)
iptables -A YAK_WORKER_FILTER -j LOG --log-prefix "YAK-WORKER-DENY: " --log-level 4

# Drop everything else
iptables -A YAK_WORKER_FILTER -j DROP

# Jump to our chain from FORWARD for worker network traffic
iptables -I FORWARD 1 -s "$WORKER_NETWORK" -j YAK_WORKER_FILTER

echo "Firewall rules configured successfully"
echo ""
echo "To view rules: iptables -L YAK_WORKER_FILTER -n -v"
echo "To view allowed IPs: ipset list yak-allowed-apis"
echo ""
echo "IMPORTANT: Run update-api-ipset.sh to populate allowed IPs"
```

**Cron job to refresh IPs (handles CDN changes):**


## Operational Procedures

### Adding a New Allowed Domain

1. Add domain to `update-api-ipset.sh`:
   ```bash
   add_domain_ips "new-api-service.com"
   ```

2. Run the script:
   ```bash
   sudo /home/yakob/yakthang/scripts/update-api-ipset.sh
   ```

3. Verify:
   ```bash
   ipset list yak-allowed-apis | grep -A5 "new-api-service"
   ```

### Viewing Denied Connection Attempts

```bash
# Real-time monitoring
journalctl -k -f | grep YAK-WORKER-DENY

# Recent denials
dmesg | grep YAK-WORKER-DENY | tail -20

# Count denials by destination
dmesg | grep YAK-WORKER-DENY | awk '{print $NF}' | sort | uniq -c | sort -rn
```

## Rollback Plan

If network filtering causes issues:

### Emergency Rollback (No Code Changes)

1. **Revert spawn-worker.sh to bridge network:**
   ```bash
   # In spawn-worker.sh line 237:
   NETWORK_MODE="bridge"  # Revert from "yak-workers"
   ```

2. **Restart any existing workers:**
   ```bash
   docker ps --filter "name=yak-worker" -q | xargs docker kill
   ```

3. **Workers will now use default bridge network** (unrestricted)

### Full Cleanup

```bash
# Stop systemd services
systemctl stop yak-worker-network.service
systemctl stop yak-worker-network-refresh.timer
systemctl disable yak-worker-network.service
systemctl disable yak-worker-network-refresh.timer

# Remove iptables rules
iptables -D FORWARD -s 172.20.0.0/16 -j YAK_WORKER_FILTER
iptables -F YAK_WORKER_FILTER
iptables -X YAK_WORKER_FILTER

# Remove ipset
ipset destroy yak-allowed-apis

# Remove Docker network (after killing workers)
docker network rm yak-workers
```

## Security Considerations

### Threat Model

**What this protects against:**
- ✅ Data exfiltration to arbitrary domains
- ✅ Workers cloning malicious repos
- ✅ Workers participating in DDoS attacks
- ✅ Workers downloading malware from internet
- ✅ Workers accessing internal network services (if Docker daemon configured properly)

**What this does NOT protect against:**
- ❌ Data exfiltration through allowed API endpoints (workers need these to function)
- ❌ Malicious LLM responses (out of scope)
- ❌ Attacks from compromised API endpoints
- ❌ Layer 7 protocol abuse (HTTP tunneling through allowed endpoints)

   - Current: Anthropic, OpenAI, models.dev, Google Gemini

2. **DNS refresh frequency:** Is 1 hour appropriate?

3. **Logging verbosity:** Should we log all denied connections?
   - Recommendation: Log denials, but with rate limiting

4. **IPv6 support:** Do we need to handle IPv6?
   - Recommendation: Support IPv6 if host has IPv6 connectivity

5. **Emergency bypass:** Should we add a --disable-network-filtering flag to spawn-worker.sh?
   - Recommendation: Add flag, but require explicit opt-in and log usage
